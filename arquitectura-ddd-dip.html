Principios SOLID

Los cinco principios fundamentales del diseño orientado a objetos que permiten crear software mantenible, escalable y flexible.

S
Single Responsibility Principle
Responsabilidad Única

Una clase debe tener una, y solo una, razón para cambiar. Cada clase encapsula una única responsabilidad o concepto del dominio.

Analogía del mundo real
En un restaurante, el chef cocina, el mesero atiende mesas y el cajero cobra. Si una sola persona hiciera todo, cualquier cambio en el menú, el servicio o los pagos afectaría a la misma persona.
C# — Violación SRP public class ReporteEmpleado { public void CalcularSalario() { // Lógica de negocio } public void GuardarEnBD() { // Persistencia } public void GenerarPDF() { // Presentación } public void EnviarEmail() { // Notificación } } // 4 razones para cambiar = violación SRP
C# — SRP Aplicado public class CalculadoraSalario { public decimal Calcular(Empleado e) { ... } } public class EmpleadoRepositorio { public void Guardar(Empleado e) { ... } } public class ReportePDF { public byte[] Generar(Empleado e){ ... } } public class NotificadorEmail { public void Enviar(string to) { ... } } // Cada clase tiene UNA razón para cambiar

O
Open/Closed Principle
Abierto/Cerrado

Las entidades de software deben estar abiertas para extensión, pero cerradas para modificación. Añadir comportamiento nuevo sin alterar código existente.

Analogía del mundo real
Un smartphone está cerrado (no modificas su hardware), pero abierto a extensión mediante apps. Agregas funcionalidad sin abrir el teléfono.
C# — Violación OCP public class CalculadoraDescuento { public decimal Calcular(string tipo, decimal monto) { if (tipo == "Regular") return monto * 0.1m; if (tipo == "Premium") return monto * 0.2m; if (tipo == "VIP") return monto * 0.3m; // Cada nuevo tipo = modificar este método return 0; } }
C# — OCP Aplicado public interface IDescuento { decimal Calcular(decimal monto); } public class DescuentoRegular : IDescuento { public decimal Calcular(decimal m) => m * 0.1m; } public class DescuentoVIP : IDescuento { public decimal Calcular(decimal m) => m * 0.3m; } // Nuevo tipo? Solo agregar nueva clase. Sin tocar las existentes.

L
Liskov Substitution Principle
Sustitución de Liskov

Los objetos de una clase derivada deben poder sustituir a los de su clase base sin alterar el correcto funcionamiento del programa.

Analogía del mundo real
Si esperas un "vehículo" para transportarte, cualquier subtipo (auto, camioneta, bus) debería funcionar. Un "vehículo de juguete" rompe el contrato porque no te puede transportar.
El problema Rectángulo/Cuadrado public class Rectangulo { public virtual int Ancho { get; set; } public virtual int Alto { get; set; } public int Area() => Ancho * Alto; } public class Cuadrado : Rectangulo { public override int Ancho { set { base.Ancho = value; base.Alto = value; } } public override int Alto { set { base.Alto = value; base.Ancho = value; } } } // Cuadrado rompe el comportamiento esperado de Rectangulo Rectangulo r = new Cuadrado(); r.Ancho = 5; r.Alto = 3; // Esperamos Area = 15, pero obtenemos 9!
Solución: abstracción común public interface IForma { int Area(); } public class Rectangulo : IForma { public int Ancho { get; } public int Alto { get; } public Rectangulo(int a, int h) { Ancho=a; Alto=h; } public int Area() => Ancho * Alto; } public class Cuadrado : IForma { public int Lado { get; } public Cuadrado(int l) { Lado=l; } public int Area() => Lado * Lado; } // Ambos cumplen el contrato de IForma correctamente

Demo interactivo: ¿Puede Cuadrado sustituir a Rectángulo?

Rectángulo

Area: 15
Cuadrado (hereda Rectángulo)

Area: 9 ≠ 15!

El Cuadrado fuerza Alto=3, haciendo Ancho=3 también. No puede sustituir al Rectángulo.

¿Por qué Cuadrado no puede heredar de Rectángulo correctamente?
Porque al cambiar una dimensión, fuerza la otra, violando el contrato del padre
Porque un cuadrado no es una forma geométrica
Porque los cuadrados no tienen área
I
Interface Segregation Principle
Segregación de Interfaces

Los clientes no deben verse forzados a depender de interfaces que no utilizan. Mejor muchas interfaces específicas que una interfaz general.

Analogía del mundo real
Un control remoto universal tiene 50 botones, pero para la TV solo usas 10. Mejor tener controles específicos: uno para TV, otro para aire acondicionado, etc.
Interfaz gorda public interface ITrabajador { void Trabajar(); void Comer(); void Dormir(); void HacerReporte(); void AsistirReunion(); } // Un Robot no come ni duerme... public class Robot : ITrabajador { public void Trabajar() { ... } public void Comer() { throw new NotImplementedException(); } // Forzado! public void Dormir() { throw new NotImplementedException(); } ... }
Interfaces segregadas public interface ITrabajable { void Trabajar(); } public interface IAlimentable { void Comer(); } public interface IDescansable { void Dormir(); } public interface IReporteable { void HacerReporte(); } public class Empleado : ITrabajable, IAlimentable, IDescansable, IReporteable { ... } public class Robot : ITrabajable, IReporteable { ... } // Robot solo implementa lo que necesita

D
Dependency Inversion Principle ESTRELLA
Inversión de Dependencias

Los módulos de alto nivel no deben depender de módulos de bajo nivel. Ambos deben depender de abstracciones. Las abstracciones no deben depender de detalles; los detalles deben depender de abstracciones.

Analogía: El enchufe eléctrico
Un electrodoméstico (alto nivel) no se conecta directamente al cableado eléctrico (bajo nivel). Ambos dependen de una abstracción: el enchufe/tomacorriente. Puedes cambiar la licuadora por una tostadora sin modificar la instalación eléctrica. El enchufe es la interfaz.
C# — Violación DIP public class PedidoService { private readonly SqlServerRepository _repo; // Dependencia directa! private readonly SmtpEmailSender _email; // Dependencia directa! public PedidoService() { _repo = new SqlServerRepository(); // Acoplado a SQL Server _email = new SmtpEmailSender(); // Acoplado a SMTP } } // No puedes testear sin base de datos real ni servidor SMTP
C# — DIP Aplicado public interface IPedidoRepository { Pedido ObtenerPorId(int id); } public interface INotificador { void Notificar(string msg); } public class PedidoService { private readonly IPedidoRepository _repo; // Abstracción! private readonly INotificador _notif; // Abstracción! public PedidoService(IPedidoRepository repo, INotificador notif) { _repo = repo; // Inyectado por DI Container _notif = notif; } } // Ahora puedes inyectar MockRepository para tests!

Domain-Driven Design — Conceptos

Diseño guiado por el dominio: una filosofía para abordar la complejidad del software alineando el código con el conocimiento del negocio. Acuñado por Eric Evans en 2003.

?
¿Qué es DDD?

DDD es un enfoque para el desarrollo de software que pone el dominio del negocio en el centro de todas las decisiones de diseño. No es un framework ni una tecnología, sino un conjunto de principios y patrones para modelar software complejo.

  • Colaboración estrecha entre desarrolladores y expertos del dominio
  • Modelo de dominio como artefacto central del proyecto
  • Código que refleja el lenguaje del negocio
  • Gestión explícita de la complejidad

Un lenguaje común, riguroso, compartido por desarrolladores y expertos del dominio. Cada término tiene un significado preciso que se refleja directamente en el código.

BC
Bounded Contexts (Contextos Delimitados)

Límites explícitos dentro de los cuales un modelo de dominio es válido. Cada contexto tiene su propio lenguaje ubicuo y sus propias reglas. El mismo concepto puede significar cosas diferentes en distintos contextos.

Ventas
"Producto" = lo que el cliente compra
Modelo de Producto en Ventas:
• Nombre, Precio, Descuento
• Disponibilidad, Categoría
• No le importa: peso, dimensiones físicas del paquete
Almacén
"Producto" = lo que se almacena
Modelo de Producto en Almacén:
• SKU, Ubicación, Peso, Dimensiones
• Stock mínimo, Fecha vencimiento
• No le importa: precio de venta, descuento
Facturación
"Producto" = línea de factura
Modelo de Producto en Facturación:
• Código, Descripción fiscal, IVA
• Unidad de medida, Código arancelario
• No le importa: stock, ubicación
Producción
"Producto" = lo que se fabrica
Modelo de Producto en Producción:
• Receta, Materiales, Tiempo de ciclo
• Máquina asignada, Orden de producción
• No le importa: precio, categoría comercial

Patrones de Context Map

Shared Kernel
Dos contextos comparten un subconjunto del modelo. Cambios requieren coordinación.
Customer-Supplier
Un contexto (supplier) provee datos al otro (customer). El customer puede negociar requerimientos.
Conformist
El contexto downstream adopta el modelo del upstream sin modificaciones. No hay negociación.
Anticorruption Layer
Capa de traducción que protege el modelo propio de modelos externos (sistemas legacy, APIs).
Open Host Service
Expone un protocolo bien definido (API pública) para que múltiples consumidores se integren.
Published Language
Formato de intercambio documentado (JSON Schema, Protobuf, XML) entre contextos.

Patrones Tácticos DDD

Los bloques de construcción del diseño guiado por el dominio: las piezas con las que modelas la lógica de negocio dentro de cada Bounded Context.

E
Entity (Entidad)
Identidad + Mutable

Objeto con identidad única que persiste a lo largo del tiempo. Aunque sus atributos cambien, sigue siendo la misma entidad.

C# public class Pedido { public Guid Id { get; } // Identidad inmutable public EstadoPedido Estado { get; private set; } public DateTime FechaCreacion { get; } public void Confirmar() { if (Estado != EstadoPedido.Borrador) throw new InvalidOperationException(); Estado = EstadoPedido.Confirmado; } }

Cambia el estado — la identidad persiste:

Pedido #A7F3
Estado: Borrador
VO
Value Object (Objeto de Valor)
Sin identidad + Inmutable

Objeto definido por sus atributos, no por una identidad. Dos VOs con los mismos valores son iguales. Son inmutables: para cambiar, se crea uno nuevo.

C# public record Dinero { public decimal Cantidad { get; init; } public string Moneda { get; init; } public Dinero Sumar(Dinero otro) { if (Moneda != otro.Moneda) throw new InvalidOperationException(); return new Dinero { Cantidad = Cantidad + otro.Cantidad, Moneda = Moneda }; } }

Igualdad por valor:

Dinero A
$100 USD
=
Dinero B
$100 USD

Mismos valores = misma entidad conceptual (sin Id)

Repository (Repositorio)
Abstracción de Persistencia

Interfaz tipo colección para acceder a agregados. La interfaz vive en el dominio, la implementación en infraestructura — esto es DIP en acción.

DS
Domain Service (Servicio de Dominio)
Lógica multi-entidad

Operación del dominio que no pertenece naturalmente a una sola entidad. Trabaja con múltiples agregados o requiere lógica que trasciende una entidad.

C# public class TransferenciaService { public void Transferir(Cuenta origen, Cuenta destino, Dinero monto) { origen.Debitar(monto); destino.Acreditar(monto); // Lógica que involucra DOS agregados } }

Algo que ocurrió

Representa algo que sucedió en el dominio y que es relevante para otras partes del sistema. Siempre en pasado: PedidoCreado, PagoRealizado, StockAgotado.

F
Factory (Fábrica)
Creación compleja

Encapsula la lógica de creación de objetos complejos (agregados con muchas dependencias). Garantiza que el objeto se cree en un estado válido.

C# public class PedidoFactory { public Pedido CrearPedido(Cliente cliente, List<Producto> items) { var pedido = new Pedido(Guid.NewGuid(), cliente.Id); foreach (var item in items) pedido.AgregarLinea(item); pedido.ValidarReglas(); return pedido; } }
SP
Specification (Especificación)
Regla de negocio encapsulada

Patrón que encapsula una regla de negocio en un objeto reutilizable. Se pueden combinar con AND, OR, NOT.

C# public class EsClientePremium : Specification<Cliente> { public override bool IsSatisfiedBy(Cliente c) => c.Tipo == TipoCliente.Premium; } // Uso combinado: var spec = new EsClientePremium().And(new TieneCreditoDisponible()); var aptos = clientes.Where(c => spec.IsSatisfiedBy(c));

TS
Ventajas para Testing

DIP permite reemplazar dependencias reales con mocks en tests unitarios. El servicio no sabe si usa la BD real o un fake.

Sin DIP — no testeable
// Necesita SQL Server real para testear var svc = new PedidoService(); // Internamente crea SqlServerRepository // No puedes aislar la lógica
Con DIP — totalmente testeable
// Mock en lugar de BD real var mockRepo = new MockPedidoRepository(); var mockNotif = new MockNotificador(); var svc = new PedidoService(mockRepo, mockNotif); // Test rápido, sin infraestructura

Quiz Final Comprehensivo
