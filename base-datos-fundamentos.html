¿Qué es una base de datos?

Es un sistema para guardar información de forma estructurada, con reglas que evitan errores y permiten consultar rápido. En la práctica, una base de datos bien diseñada reduce retrabajo, mejora reportes y evita inconsistencias como clientes duplicados o saldos incorrectos.

¿Para qué sirve?

  • Centralizar datos en una sola fuente de verdad
  • Evitar duplicidad e inconsistencias
  • Soportar operación diaria y decisiones
  • Mantener historial y trazabilidad

DB vs DBMS

DB: los datos (tablas, registros, relaciones). DBMS: el motor que permite crear, consultar, asegurar y respaldar esos datos (MySQL, PostgreSQL, SQL Server, Oracle).

Tipos principales

  • Relacional (SQL): ideal para procesos transaccionales y reglas estrictas
  • NoSQL: útil cuando cambian mucho los esquemas o se requiere alta escalabilidad horizontal
  • Data warehouse: orientado a análisis histórico y BI

Ejemplo real: por qué importa modelar bien

Sin diseño

Una sola tabla con ventas, cliente, vendedor y producto repetidos en cada fila. Resultado: errores frecuentes al actualizar y reportes lentos.

Con diseño relacional

Tablas separadas para clientes, productos, ventas y detalle_venta. Menos duplicación, mejor control y consultas más claras.

Conceptos básicos que debes dominar

Tabla

Representa una entidad del negocio. Ejemplo: la tabla clientes almacena solo datos de clientes, no de productos ni pagos.

Fila (registro)

Es un elemento específico. Si la tabla es clientes, cada fila es un cliente distinto.

Columna (atributo)

Define qué dato guardar y con qué tipo: nombre VARCHAR(100), fecha_registro DATE, saldo DECIMAL(10,2).

Tipo de dato

Controla calidad de datos y rendimiento. Elegir bien tipos evita errores y desperdicio de almacenamiento.

Clave primaria (PK)

Identificador único por fila. Debe ser estable y no cambiar con frecuencia.

Clave foránea (FK)

Conecta tablas. Una venta referencia al cliente que la realizó.

Flujo de trabajo de datos

1) Levantar reglas del negocio
2) Diseñar entidades y relaciones
3) Crear tablas y restricciones
4) Cargar y validar datos
5) Consultar y medir rendimiento
6) Respaldar y monitorear

Errores típicos de principiante

Guardar todo como texto

Complica filtros, sumas y validaciones. Usa tipos correctos.

No definir PK/FK

Se pierde integridad y aparecen datos huérfanos.

No validar nulos

Campos críticos vacíos rompen lógica de negocio y reportes.

Modelo relacional

Organiza datos en tablas con relaciones explícitas. La idea central es simple: cada dato se guarda una sola vez donde corresponde, y se conecta con llaves para consultar todo junto sin duplicar.

RelaciónDescripciónEjemplo
1 a 1Un registro se relaciona con uno del otro ladoUsuario ↔ Perfil
1 a NUn registro se relaciona con muchosCliente ↔ Pedidos
N a NMuchos a muchos mediante tabla intermediaEstudiante ↔ Curso

Claves y restricciones

PK

Clave principal única. Ejemplo recomendado: id_cliente entero autoincremental o UUID.

FK

Clave foránea para relacionar tablas y evitar registros huérfanos. Ejemplo: venta sin cliente no debería existir.

UNIQUE

Evita duplicados en valores que deben ser únicos: correo, cédula, código.

NOT NULL

Obliga a ingresar datos críticos. Ejemplo: una orden sin fecha no debería guardarse.

CHECK

Valida reglas: precio > 0, estado IN ('A','I').

DEFAULT

Asigna valores por defecto para reducir errores manuales.

Mini modelo de ejemplo (ventas)

clientes(id_cliente PK, nombre, correo UNIQUE) productos(id_producto PK, nombre, precio CHECK (precio > 0)) ventas(id_venta PK, id_cliente FK, fecha) detalle_venta(id_detalle PK, id_venta FK, id_producto FK, cantidad CHECK (cantidad > 0), precio_unitario)

SQL básico (CRUD)

SQL se divide en familias: DDL (estructura), DML (datos), DCL (permisos) y TCL (transacciones). Primero domina DDL + DML.

CREATE TABLE clientes ( id INT PRIMARY KEY, nombre VARCHAR(100) NOT NULL, correo VARCHAR(150) UNIQUE ); INSERT INTO clientes (id, nombre, correo) VALUES (1, 'Ana Pérez', 'ana@correo.com'); SELECT id, nombre, correo FROM clientes WHERE nombre LIKE 'Ana%'; UPDATE clientes SET correo = 'ana.perez@correo.com' WHERE id = 1; DELETE FROM clientes WHERE id = 1;

¿Qué es DDL, DML y DCL?

TipoPara qué sirveComandos típicos
DDL (Data Definition Language)Define o cambia la estructura de la base de datosCREATE, ALTER, DROP, TRUNCATE
DML (Data Manipulation Language)Inserta, consulta, modifica o elimina datosINSERT, SELECT, UPDATE, DELETE
DCL (Data Control Language)Controla permisos y seguridad de accesoGRANT, REVOKE
-- DDL: crear estructura CREATE TABLE productos ( id INT PRIMARY KEY, nombre VARCHAR(120) NOT NULL, precio DECIMAL(10,2) CHECK (precio > 0) ); -- DML: manipular datos INSERT INTO productos (id, nombre, precio) VALUES (1, 'Resina PE', 25.50); UPDATE productos SET precio = 26.00 WHERE id = 1; SELECT id, nombre, precio FROM productos; -- DCL: controlar acceso GRANT SELECT, INSERT ON productos TO usuario_ventas; REVOKE INSERT ON productos FROM usuario_ventas;

JOIN explicado con lógica de negocio

SELECT v.id_venta, c.nombre, v.fecha FROM ventas v INNER JOIN clientes c ON c.id_cliente = v.id_cliente; -- INNER JOIN: solo ventas con cliente válido. -- LEFT JOIN: trae todos los clientes, tengan o no ventas.

Consultas útiles

JOIN

Une tablas para responder preguntas reales: “¿qué cliente compró qué producto?”.

GROUP BY

Agrupa datos para análisis: ventas por mes, por vendedor o por categoría.

ORDER BY

Ordena resultados ascendente o descendente.

LIMIT/TOP

Restringe cantidad de filas devueltas.

Buenas prácticas de consulta

Evita SELECT *

Pide solo columnas necesarias para reducir tráfico y costo de ejecución.

Filtra temprano

Usa WHERE para reducir filas antes de agrupar/ordenar.

Normalización (en términos simples)

Normalizar es organizar datos para minimizar redundancia y anomalías al insertar, actualizar o eliminar.

Forma NormalQué evitaIdea principal
1FNCampos multivalorDatos atómicos, una cosa por celda
2FNDependencias parcialesTodo atributo depende de la PK completa
3FNDependencias transitivasAtributos no clave dependen solo de la PK

Anomalías que evita la normalización

Inserción

No deberías crear una venta para poder registrar por primera vez un producto.

Actualización

Cambiar el correo de un cliente en 20 filas distintas genera riesgo de inconsistencias.

Eliminación

Borrar una venta no debe eliminar accidentalmente la información maestra del cliente.

Duplicación

Menos repetición de datos reduce errores y espacio usado.

¿Siempre normalizar al máximo?

No necesariamente. En analítica o reportes se usa desnormalización controlada para acelerar lecturas, siempre evaluando costo de mantenimiento y consistencia.

Transacciones y propiedades ACID

Una transacción agrupa operaciones relacionadas para que el sistema no quede “a medias”. Es crítica en banca, inventario, pagos y cualquier proceso sensible.

A — Atomicidad

Todo o nada. Si falla una parte, se revierte todo.

C — Consistencia

Los datos pasan de un estado válido a otro válido.

I — Aislamiento

Transacciones concurrentes no se “pisan” entre sí.

D — Durabilidad

Confirmado el commit, el dato persiste ante fallos.

Ejemplo básico de transacción

BEGIN TRANSACTION; UPDATE cuentas SET saldo = saldo - 100 WHERE id = 1; UPDATE cuentas SET saldo = saldo + 100 WHERE id = 2; COMMIT; -- o ROLLBACK si ocurre error

Read CommittedEquilibrio común en OLTPLecturas no repetibles
Repeatable ReadLecturas estables por transacciónPhantoms según motor

Índices y rendimiento

Un índice funciona como el índice de un libro: evita leer todo. Ayuda en búsquedas y joins, pero cada escritura debe actualizarlo, por eso hay que diseñarlo con criterio.

Buena prácticaRazón
Indexar PK y FKMejora joins y filtros comunes
Indexar columnas de búsqueda frecuenteReduce full scans
Evitar exceso de índicesMás costo de escritura y mantenimiento
Revisar planes de ejecuciónDiagnóstico real de cuellos de botella

Cuándo un índice sí/no ayuda

Sí ayuda

Filtros por columnas selectivas, joins por FK y ordenamientos frecuentes.

No tanto

Columnas con pocos valores distintos (ej. estado A/I) o tablas muy pequeñas.

Compuesto

El orden de columnas importa: diseña según filtros más usados.

Métrica clave

Valida con plan de ejecución y tiempo real, no por intuición.

Administración básica de BD

Administrar una BD es garantizar disponibilidad, seguridad, rendimiento y recuperación ante fallos.

Seguridad

  • Principio de mínimo privilegio
  • Roles y permisos por función
  • Auditoría de accesos

Respaldos

  • Backups completos + incrementales
  • Pruebas periódicas de restauración
  • RPO/RTO definidos

Mantenimiento

  • Actualización de estadísticas
  • Reorganización/reconstrucción de índices
  • Limpieza de datos obsoletos

Monitoreo

  • CPU, memoria, IOPS
  • Queries lentas
  • Bloqueos y deadlocks

Plan mínimo operativo recomendado

Backups diarios + prueba mensual de restore
Monitoreo de queries lentas
Revisión semanal de bloqueos/deadlocks
Parcheo y actualización controlada
Documentar RPO y RTO del negocio
