🖥️ Fundamentos de Sistemas Operativos

Antes de Linux y cloud, todo parte de entender qué hace un sistema operativo y cómo administra recursos: procesos, memoria, archivos, dispositivos y seguridad.

🧭 ¿Qué es un SO? (5 funciones base)

1) Gestión de procesos

Crea, planifica y finaliza procesos/hilos para usar la CPU de forma eficiente y justa.

  • Planificación (scheduler)
  • Conmutación de contexto
  • Sincronización e IPC

2) Gestión de memoria

Asigna RAM y memoria virtual a cada proceso con aislamiento y protección.

  • Memoria física y virtual
  • Paginación y swapping
  • Protección por espacios de direcciones

3) Sistema de archivos

Organiza datos en estructuras persistentes y controla permisos de acceso.

  • Directorios y metadatos
  • Permisos y propietarios
  • Lectura/escritura con buffer

4) Entrada/Salida (E/S)

Abstrae dispositivos físicos (teclado, disco, red, impresora) vía drivers.

  • Controladores (drivers)
  • Interrupciones
  • Colas y buffers de I/O

5) Seguridad y usuarios

Autentica usuarios, aplica permisos y mantiene aislamiento entre procesos.

  • Cuentas y grupos
  • Control de acceso
  • Auditoría y trazabilidad

⚙️ Tipos de proceso y estado de proceso

Tipos de proceso

  • CPU-bound: consume más CPU que I/O (cálculo intensivo).
  • I/O-bound: espera disco/red con frecuencia.
  • Foreground: ligado a terminal/interfaz activa.
  • Background / Daemon: corre en segundo plano como servicio.
  • Batch: ejecutado por cola o planificador.

Estados del proceso

Nuevo
→
Listo
→
Ejecución
→
Bloqueado
→
Terminado
  • El scheduler decide cuándo un proceso pasa de Listo a Ejecución.
  • Un proceso pasa a Bloqueado al esperar I/O o evento externo.
  • Con interrupciones y quantum vuelve a Listo.

🧮 Asignación de memoria

Técnicas principales

  • Particiones fijas: simples, pero con fragmentación interna.
  • Particiones variables: más flexibles, fragmentación externa.
  • Paginación: divide memoria en páginas/marcos, base de la memoria virtual.
  • Segmentación: organiza por bloques lógicos (código, datos, stack).
  • Memoria virtual: usa disco como extensión de RAM (swap).

Criterios de rendimiento

CriterioQué evaluar
FragmentaciónEspacio desaprovechado (interna/externa)
LatenciaTiempo de acceso y fallos de página
ThroughputProcesos completados por unidad de tiempo
AislamientoProtección entre espacios de memoria

📂 Jerarquía del Sistema de Archivos (FHS)

  • / — Raíz del sistema, todo parte de aquí
    • /bin — Binarios esenciales (ls, cp, mv, cat)
    • /boot — Kernel, initramfs, GRUB
    • /dev — Dispositivos como archivos (sda, tty, null)
    • /etc — Configuración del sistema (fstab, passwd, nginx/)
    • /home — Directorios personales de usuarios
    • /opt — Software de terceros opcional
    • /proc — Sistema de archivos virtual (info del kernel en vivo)
    • /root — Home del superusuario
    • /tmp — Archivos temporales (se limpian al reiniciar)
    • /usr — Programas y librerías de usuario
      • /usr/bin — Binarios de usuario
      • /usr/lib — Librerías compartidas
      • /usr/share — Datos independientes de arquitectura
    • /var — Datos variables (logs, mail, spool)
      • /var/log — Logs del sistema y aplicaciones
      • /var/www — Raíz web por defecto

🔐 Sistema de Permisos Unix

permisos.sh
# Formato: tipo | dueño | grupo | otros
-rwxr-xr-- 1 jefferson devs 4096 mar 23 10:00 script.sh

# Desglose:
- → archivo regular (d = directorio, l = enlace)
rwx → dueño: leer + escribir + ejecutar (7)
r-x → grupo: leer + ejecutar (5)
r-- → otros: solo leer (4)

$ chmod 755 script.sh # rwxr-xr-x
$ chmod u+x archivo # agregar ejecución al dueño
$ chown jefferson:devs archivo # cambiar dueño y grupo
$ chmod 600 id_rsa # rw------- (solo dueño, clave SSH)

☁️ Cloud Computing

La nube transformó la infraestructura: de comprar servidores a consumir recursos como servicio, bajo demanda, con escalabilidad global.

📊 Modelos de Servicio Cloud

🏗️ IaaS — Infrastructure as a Service

Infraestructura virtualizada bajo demanda. Tú gestionas desde el SO hacia arriba.

EC2Azure VMs GCEDigitalOcean

  • Proveedor: virtualización, red, storage, DC
  • Ideal para migración lift-and-shift

⚙️ PaaS — Platform as a Service

Plataforma lista para desplegar código. El proveedor gestiona la infraestructura subyacente.

HerokuAzure App Service Google App EngineElastic Beanstalk
  • Foco en el código, no en infra
  • Tú: aplicación y datos
  • Proveedor: SO, middleware, runtime
  • Escalado automático incluido

📦 SaaS — Software as a Service

Aplicaciones listas para usar vía navegador. Cero gestión de infraestructura.

GmailSlack SalesforceOffice 365
  • Solo consumir el servicio
  • Tú: datos y configuración
  • Proveedor: absolutamente todo lo demás
  • Modelo de suscripción mensual/anual

Los contenedores empaquetan aplicaciones con todas sus dependencias, garantizando que funcionen igual en cualquier entorno: desarrollo, testing y producción.

📦 Conceptos Clave de Docker

📋 Dockerfile

Receta de texto que define cómo construir una imagen paso a paso. Cada instrucción crea una capa.

FROMRUNCOPYEXPOSECMD

💿 Imagen (Image)

Template inmutable de solo lectura con el SO base, dependencias y la aplicación. Se versiona con tags.

Capas (Layers)RegistryTags

📦 Contenedor

Instancia en ejecución de una imagen. Tiene su propio filesystem, red y procesos aislados.

EfímeroStatelessAislado

📝 Anatomía de un Dockerfile

Dockerfile
# 1. Imagen base — Elegir la más ligera posible
FROM node:20-alpine

# 2. Directorio de trabajo dentro del contenedor
WORKDIR /app

# 3. Copiar archivos de dependencias primero (cache de capas)
COPY package*.json ./
RUN npm ci --only=production

# 4. Copiar el código fuente
COPY . .

# 5. Exponer puerto (documentación)
EXPOSE 3000

# 6. Usuario no-root (seguridad)
USER node

# 7. Comando de inicio
CMD ["node", "server.js"]

🧅 Capas de una Imagen Docker

CMD ["node", "server.js"]0 B
COPY . . (código fuente)~2 MB
EXPOSE 3000 / USER node0 B
RUN npm ci (node_modules)~85 MB
FROM node:20-alpine (SO + Node)~180 MB

↑ Las capas inferiores se cachean. Ordenar de menos a más cambiante optimiza el build.

🔨 Build & Lint

🧪 Testing Automatizado

JestCypressxUnit

🐳 Build & Push de Imagen

DockerECRTrivy

🌍 Deploy a Staging

Despliegue automático al entorno de staging. Pruebas de humo y validación de integración con servicios reales.

ArgoCDHelmTerraform

🚀 Deploy a Producción

Aprobación manual o automática. Estrategia de despliegue: blue-green, canary o rolling update. Rollback automático si hay errores.

🎯 Estrategias de Despliegue

🔵🟢 Blue-Green

Dos entornos idénticos. Se redirige el tráfico del blue (actual) al green (nuevo) de golpe. Rollback instantáneo volviendo al blue.

  • Pro: Rollback inmediato, zero-downtime
  • Con: Costo doble de infraestructura

🐤 Canary

Se envía un pequeño % del tráfico a la versión nueva. Si las métricas son buenas, se aumenta gradualmente hasta el 100%.

  • Pro: Detección temprana de errores
  • Con: Más complejo de implementar

🔄 Rolling Update

Las instancias se actualizan una por una (o en lotes). Siempre hay instancias activas sirviendo tráfico.

  • Pro: Eficiente en recursos, nativo de K8s
  • Con: Ambas versiones coexisten temporalmente
